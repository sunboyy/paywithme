## Dependency notes

- No hard dependencies between these four — they touch disjoint files. Suggested
  order is by leverage/risk: 003 (protects the core money path) is the most
  valuable; 001 and 002 are quick wins; 004 is a small UX improvement.
- 002 adds a Zod refine rejecting `amountTotal === 0`. 003's test inputs all use
  positive amounts, so 002 does not affect 003 even if both land.
- **Plan 004 scope was corrected during vetting.** The original direction idea
  ("let a member who left see their own outstanding balance") is unreachable:
  `userHasGroupAccess` (`src/lib/server/groups.ts`) requires
  `isNull(members.deactivatedAt)`, so a soft-deactivated user is 404'd out of every
  group page. The plan instead labels inactive members in the balances list that
  _active_ members see — the reachable, valuable version. Do not change the access
  rule.

## Findings considered and rejected (so they aren't re-audited)

- **CSRF on form actions** — SvelteKit enables origin-check CSRF protection by
  default (`csrf.checkOrigin`); nothing in this repo disables it. By-design, not a gap.
- **Missing `member_id` indexes on `transaction_payers`/`transaction_shares`** —
  the balances query (`src/lib/server/balances.ts:69-88`) filters/joins on
  `transaction_id` + `transactions.group_id` (both indexed); `member_id` is only
  in the SELECT list, so an index on it wouldn't change the query plan. Mis-analysis.
- **N+1 in transaction detail / group-list caching** — detail issues a fixed set
  of batched queries (not per-item); caching adds invalidation risk for marginal gain.
- **Latent IDOR in invite/members** — the DB-layer `WHERE groupId` guard makes
  current code safe; only a hypothetical future refactor risk.
- **God files** (`src/lib/server/transactions.ts` ~1159 lines,
  `src/lib/components/TransactionForm.svelte` ~1329 lines) — real size, but
  well-tested and well-commented; an L-effort, MED-risk refactor with thin upside
  on a finished build. Not worth a plan unless explicitly requested.
- **Dependency advisories / bleeding-edge majors** (Vite 8, ESLint 10, TS 6) —
  transitive dev-chain only; the version choice is deliberate.
- **`hooks.server.ts` swallows session-resolution errors** — real, but the inline
  comment shows it's a deliberate "never 500 the whole app" tradeoff; re-throwing
  would 500 logged-in users instead. Low leverage; left as-is.
